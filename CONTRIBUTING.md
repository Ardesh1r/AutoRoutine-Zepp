# Contributing to AutoRoutine

Thank you for your interest in contributing to AutoRoutine! We welcome contributions from the community.

## Code of Conduct

Be respectful, inclusive, and constructive in all interactions.

## Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/yourusername/AutoRoutine-Zepp.git`
3. Create a feature branch: `git checkout -b feature/your-feature`
4. Install dependencies: `npm install`
5. Build and test: `npm run build`

## Development Workflow

### Building
```bash
npm run build          # Build the project
npm run simulator      # Run simulator
npm run preview        # Preview on device
npm run log           # View device logs
```

### Code Style
- Use consistent indentation (2 spaces)
- Follow existing code patterns
- Add comments for complex logic
- Keep functions focused and small

### Testing
- Test new features on actual device when possible
- Test on multiple device models
- Check battery impact
- Verify storage usage

## Contribution Types

### Bug Fixes
1. Create an issue describing the bug
2. Reference the issue in your PR
3. Include steps to reproduce
4. Add test cases if applicable

### Features
1. Discuss in an issue first
2. Get feedback before implementing
3. Keep features focused and minimal
4. Add documentation

### Documentation
1. Fix typos and clarify explanations
2. Add examples and use cases
3. Update API documentation
4. Improve architecture docs

### Examples
1. Add sample rules
2. Create tutorial content
3. Share integration examples
4. Document common patterns

## Pull Request Process

1. Update documentation if needed
2. Add meaningful commit messages
3. Reference related issues
4. Describe changes clearly
5. Test thoroughly before submitting
6. Be responsive to feedback

## Commit Messages

Use clear, descriptive commit messages:
```
feat: Add new trigger type for activity metrics
fix: Resolve cooldown not preventing duplicate notifications
docs: Update rule configuration examples
refactor: Simplify sensor data processing
test: Add unit tests for rule evaluation
```

## Reporting Issues

Include:
- Device model and Zepp OS version
- Steps to reproduce
- Expected vs actual behavior
- Screenshots if applicable
- Device logs if relevant
